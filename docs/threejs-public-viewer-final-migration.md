# Three.js Public Viewer Final Migration

## 1. What changed

The public panorama scene route now uses the Three.js viewer by default.

Opening this route shows Three.js:

```text
/tour/[tourSlug]/[floorSlug]/[sceneSlug]
```

The old Pannellum public viewer still exists as an emergency fallback:

```text
/tour/[tourSlug]/[floorSlug]/[sceneSlug]?viewer=pannellum
```

Old Three.js test links still work too:

```text
/tour/[tourSlug]/[floorSlug]/[sceneSlug]?viewer=three
```

`viewer=three` is no longer generated by public frontend links because Three.js is now the normal public viewer.

## 2. Why the public viewer now uses Three.js

Three.js gives the public viewer direct control over the panorama camera, hotspot projection, transitions, and future visual effects. This migration moves only the public viewing experience to Three.js. The CMS data stays the same, so existing tours, scenes, hotspots, floor maps, and draft preview links continue to use the same stored fields.

## 3. What Pannellum parts remain untouched

Pannellum remains in the project.

The migration does not change:

- Payload collections or schemas.
- Backend data fetching logic beyond the public viewer mode flag.
- Admin panorama validation preview.
- Admin hotspot picker.
- Pannellum dependencies in `package.json`.
- The `PannellumViewer` component.

Pannellum can still be tested publicly with `?viewer=pannellum`.

## 5. Portal navigation algorithm

When a scene or portal hotspot is clicked, the Three.js viewer reads:

- `targetScene.slug`
- `targetFloor.slug`, if it exists
- the clicked hotspot `pitch` and `yaw`, if they exist

If `targetScene.slug` is missing, the viewer ignores the click. In development, it logs a warning so the CMS data can be checked.

If `targetFloor.slug` is missing or matches the current floor slug, the viewer treats the click as same-floor navigation.

If `targetFloor.slug` exists and differs from the current floor slug, the viewer treats the click as cross-floor navigation.

## 6. Same-floor navigation

For same-floor portal navigation:

1. The viewer finds the target scene in the current `floorScenes` array.
2. If the scene is not found, it logs a development warning.
3. If `targetFloor` was missing, the warning explains that the hotspot may be missing `targetFloor` for a cross-floor portal.
4. The page does not reload.
5. The camera animates toward the clicked portal direction.
6. At the midpoint, the active scene slug changes.
7. `TourViewer` receives `onSceneChange(targetSlug)`.
8. The overlay and sidebar use the new active scene data.
9. The camera finishes by focusing into the target scene initial yaw, pitch, and HFOV.

The hotspot sidebar also uses the Three.js viewer API for same-floor scene changes when Three.js is active.

## 7. Cross-floor navigation

For cross-floor portal navigation:

1. The camera animates toward the clicked portal direction.
2. The viewer builds a route URL:

```text
/tour/[tourSlug]/[targetFloorSlug]/[targetSceneSlug]
```

3. If draft preview is active, it preserves:

```text
?draft=true
```

4. It navigates with `window.location.assign(...)`.

The public links no longer add `viewer=three`. If the current session is using `?viewer=pannellum`, generated links preserve `viewer=pannellum` for fallback testing.

## 8. Google Maps-style transition

The new helper file is:

```text
src/components/tour/three/threeCameraTransition.ts
```

It provides:

- `easeInOutCubic(t)`
- `interpolateNumber(start, end, t)`
- `getShortestYawDelta(startYaw, endYaw)`
- `interpolateYaw(startYaw, endYaw, t)`
- `animateCameraTransition(...)`
- `animateCameraExit(...)`
- `animateCameraFocus(...)`

Same-floor transitions work like this:

1. Read the current camera state.
2. Zoom out slightly by increasing HFOV.
3. Rotate toward the clicked portal direction, or the target scene initial direction if the hotspot direction is unavailable.
4. Switch the active scene at the midpoint.
5. Place the camera near the target scene initial direction.
6. Smoothly focus into the target scene initial yaw, pitch, and HFOV.

Cross-floor transitions use the same outgoing camera travel, then navigate to the route URL.

If the user prefers reduced motion, the complex travel animation is skipped and the camera state changes directly.

## 9. Scene rotation and pitch/yaw

CMS pitch and yaw values are still stored exactly as authored. The migration does not rewrite database data.

The Three.js public viewer applies `scene.rotation` only when it converts stored CMS coordinates into display coordinates. This happens in:

```text
src/components/tour/three/threePanoramaMath.ts
```

The rule is:

```text
displayYaw = storedYaw + scene.rotation
```

Pitch is not rotated by this field. Pitch still means vertical angle:

- positive pitch looks upward
- negative pitch looks downward

Yaw is the horizontal direction around the panorama. `scene.rotation` shifts the horizontal yaw origin for the whole scene, so the initial camera direction and hotspots stay aligned with a panorama image that has an offset.

The same helper is used for:

- initial camera direction
- hotspot display position
- info hotspot focus
- portal transition direction

This keeps rotation behavior consistent without changing Payload data or schema.

## 10. Info hotspot focus

Info hotspots now focus the Three.js camera before opening their dialog.

The behavior is:

1. User clicks an info hotspot.
2. The hotspot click stops propagation so it does not start panorama dragging.
3. The camera animates to the hotspot pitch and yaw.
4. The camera uses a tighter HFOV for focus.
5. The existing info dialog opens after the focus completes.

The hotspot sidebar also focuses the camera before showing info content. Because the sidebar did not previously own the hotspot button dialog, it now opens a small matching dialog using the same hotspot text and info content.

## 11. Draft preview navigation

The draft preview route still starts here:

```text
/tour/[tourSlug]/preview
```

It redirects to the default draft scene with:

```text
?draft=true
```

If fallback testing is active, this is preserved:

```text
/tour/[tourSlug]/preview?viewer=pannellum
```

That redirects to:

```text
/tour/[tourSlug]/[floorSlug]/[sceneSlug]?draft=true&viewer=pannellum
```

Inside the Three.js public viewer, cross-floor navigation also preserves `draft=true`.

## 12. How to test

Test these routes:

```text
/tour/[tourSlug]
/tour/[tourSlug]/preview
/tour/[tourSlug]/[floorSlug]/[sceneSlug]
/tour/[tourSlug]/[floorSlug]/[sceneSlug]?viewer=three
/tour/[tourSlug]/[floorSlug]/[sceneSlug]?viewer=pannellum
/tour/[tourSlug]/[floorSlug]/[sceneSlug]?draft=true
```

Check these behaviors:

- The normal public scene route opens Three.js.
- `?viewer=pannellum` opens the old Pannellum public viewer.
- The landing page Start Tour button opens the default floor and scene.
- Same-floor portal hotspots change scenes without a page reload.
- Cross-floor portal hotspots navigate to the right route.
- Floor map navigation still opens the requested scene.
- Hotspot sidebar scene navigation still works.
- The home button still works.
- Draft preview keeps `draft=true`.
- Info hotspots focus the camera before opening content.
- Visible public viewer UI text is English.
- Scenes with non-zero `rotation` keep their initial view and hotspots aligned.

## 13. Known limitations and future improvements

- Same-floor Three.js scene switching currently starts loading the new panorama at the scene switch point. A future improvement could preload the target panorama before the midpoint.
- The old transition overlay system remains available for Pannellum and transition previews, but the Three.js scene travel now uses camera animation as the main transition.
- The Pannellum fallback remains public for emergency testing. It can be removed later only after admin/editor dependencies are handled separately.
